Para desarrollar aplicaciones en Salesforce, necesita las siguientes herramientas.
1. Un entorno de desarrollo de Salesforce, también llamado organización.
2. La interfaz de línea de comandos (CLI) de Salesforce para gestionar el ciclo de vida del
 desarrollo de la aplicación.
3. Visual Studio Code (VS Code) instalado


COMMAND LINE INTERFACE (CLI)

Es la herramienta de línea de comando para trabajar con Salesforce Platform.
puede crear entornos para desarrollos y pruebas de forma sencilla. Además, Salesforce CLI tiene
comandos para sincronizar código fuente entre sus organizaciones y sistemas de control de 
versiones, y ejecutar pruebas de unidades.

Instalar la Command Line Interface:
 	Descargar: 	https://developer.salesforce.com/tools/salesforcecli

sf --versión -> Versifica que el cli esta instalado.

sf update -> Actualizo todo lo relacionado a Salesforce


CONECTAR VSC CON UN PLAYGROUND

Es necesario que en vsc se tenga instalada la extensión Salesforce Extension Pack (expanded).

Crear un nuevo proyecto de Salesforce (desde VS Code): *En vsc* ->  Ctrl + Shift + P -> escribir 
	'SFDX: Create Project with Manifest' -> estándar -> Asigna un nombre al proyecto ->
 	Seleccione una carpeta en su dispositivo para guardar el proyecto.

Autorizar determinado playgroud desde cli: Usamos el comando: sf org login web -> Después nos
 	pedirá ingresar un usuario y contraseña. Este usuario y contraseña deben ser los que 	
	específicamente tiene cada playground, los cuales son diferentes a las credenciales que
 	usarmos para iniciar sesión en Salesforce.
 	Para determinar el usuiairo y contraseña de un playground deberemos restablecer la
 	contraseña de determinado playground.

Restablecer la contraseña para su organización de trailhead playgroud: La integración entre
 	Trailhead y la organización de Playground hace que la contraseña predeterminada sea
 	inaccesible. Necesita una contraseña para acceder a la organización desde fuera de
 	Trailhead, por ejemplo desde herramientas de desarrollador, como Salesforce CLI y
 	VS Code.
 	Clic en Iniciador de aplicación para iniciar App Launcher -> buscar y hacer clic en
 	Playground Starter -> click en apartado get your login credentials -> en esta parte
 	aparecerá un correo, el cual deberemos tomar (este vendrá a ser el ussername que
 	utilizaremos en próximos comandos) -> después presionar reset my password.

sf org list -> Muestra todas las organizaciones de Salesforce que tienes autorizadas en tu
 	Salesforce CLI.
 	También mostrara cual playground tenemos seleccionado por defecto.

sf org open -> Abre la organización por defecto.
 	- sf org open -o [nombreAlias/username] -> Abre determinada organización.

sf config set target-org=[username] -> Define cuál organización de Salesforce usará la CLI por
 	defecto, a nivel global, cuando no se especifique otra.

*Dentro de la carpeta que representa a un proyecto*
sf project retrieve start -x manifest/package.xml -> Para traer (recuperar) metadatos desde tu
 	org de Salesforce
 	Todo lo que se recupera se guarda en: force-app/main/default/.
 		- Reglas de validación: objects/Contact/validationRules/
 		- flows/
 		- layouts/
 		- objects
 	El archivo package.xml es un archivo que existe en nombreProyecto/manifest/package.xml.
 	En el archivo package.xml se debe especificar que se quiere descargar, se debe agregar
 	asi:
 		<types>
                  <members>*</members>
                  <name>tipoObjeto</name>
    		</types>
 	Donde el tipo objeto seria los flows, objetos, etc.
 

A la hora de desarrollar en Salesforce Platform, existen numerosos lenguajes de programación que puede usar, por ejemplo:
 	- El marco Lightning Web Components: un marco de trabajo de desarrollo de interfaz
 		de usuario basada en JavaScript.
 	- Apex: lenguaje de programación creado por Salesforce con una sintaxis similar a Java.

Proporciona únicamente los elementos necesarios para ejecutarse correctamente en los navegadores admitidos en Salesforce.

	pueda mostrar a su componente hijo.
	- onproductselected={handleProductSelected} -> 

<template for:each={nombreArregloX} for:item="elementoX">

Comunicaciones entre componentes -----------------------------

Los componentes se pueden comunicar entre un padre y un hijo en ambos sentidos.

     Comunicación ascendente 

Para llevar a cabo una comunicación hacia arriba se usan eventos y/o agentes de escucha de eventos.
El componente secundario envía el evento y el principal lo escucha. 

*En la vista del padre*
<c-nombreComponenteHijoX onNombreEventoPadreY={nombreEventoPadreY}></c-nombreComponenteHijoX> -> 	Se llama al componenete hijo y como atributo de la etiqueta se le pasa el nombre del 
	evento del padre.
	Por nomeclatura el nombre del atributo se le antepone el prefijo on.

*En el archivo js del padre*
nombreEventoPadreY() {
   // n instrucciones
} 

     Comunicación descendente

La información se puede comunicar en sentido descendente mediante propiedades públicas y métodos públicos.

*En el archivo js del hijo*
@api nombreAtributo; -> Para recibir valores.

*En el archivo html del padre*
<c-nombreComponenteHijoX nombre-atributo={valor}></c-nombreComponenteHijoX> -> Para mandar 	valores desde el padre al hijo.
	Observese que en la vista, el nombre del atributo usa la nomeclatura snake-case y en
	js se ocupa camelCase.

Incorporar datos de Salesforce a nuestros componentes -----------------

El servicio "wire" incorpora datos en su aplicación.
Este servicio forma parte de nuestra plataforma y proporciona una transmisión de datos.

@wire(tipoAdaptador, { parametros })
nombreAtributo; -> Para llenar automáticamente una propiedad de tu componente con los datos que 
	devuelve el adaptador (UI API o Apex).	
	El atributo contendrá dos atirubots el primero (data) contiene la infor solicitada y 
	el segundo (error) un posible mensaje de errro.
	Tipos de adaptadores:
	Cada uno de estos se deben importar desde diferentes paquetes.
	Adaptadores para LDS ------------------
	- getRecord -> Para traerse en un registro individual.
	- getRecordUi -> Para metadatos del registro.
	- getObjectInfo -> Metadatos del objeto.
	Valores para el segundo parámetro:
	- Si se uso getRecord: Se debe pasar un objeto con dos atributos los cuales define QUÉ 
		datos traes y CÓMO los traes.
		En el primer atributos se coloca el id del registro (Es obligatorio) y en el 
		segundo atributo se especifica un array de campos que deseamos traernos del 
		registro.

@wire(tipoAdaptador, { parametros })
wiredNombreObjeto({data, error}) {
 // logica
}
nombreAtributo; -> Se usa igualito que el anterior pero se anexa una función entre el atributo
	y la anotación para agregar lógica a los datos.

Decoradores ----------------------------

Los decoradores se suelen usar en JavaScript para modificar el comportamiento de una propiedad o función.
Para usar un decorador, impórtelo desde el módulo lwc y colóquelo delante de la propiedad o función.
una propiedad o función solo puede tener un decorador y no mas. *

@api -> Hace que una propiedad o método sea público.
	Usado para:
	Recibir datos desde un componente padre
	Permitir que el padre llame métodos del hijo
	Configurar el componente desde fuera.

@track -> Se usa para hacer reactivos los cambios en objetos o arreglos, de modo que la UI se 
	vuelva a renderizar cuando cambian.
	Hoy en día casi no se usa, porque LWC ya maneja reactividad automática en muchos 
	casos. **

@wire se usa para obtener datos reactivos automáticamente desde Apex o desde Lightning Data 
	Service (LDS) sin escribir lógica imperativa.
	Para conectar un componente LWC a una fuente de datos de Salesforce y mantener la UI 
	sincronizada automáticamente cuando los datos cambian.

Archivo de configuración ---------------------------

El archivo que tiene la extensión .js-meta.xml, es el archivo de configuración de un 
componente. Este archivo proporciona metadatos para Salesforce, incluida la configuración de 
diseño de los componentes que se van a usar en Lightning App Builder

Configuraciones de este archivo:

apiVersion -> vincula el componente a una versión de API de Salesforce.

isExposed -> Si vale false el componente no se expone en Lightning App Builder ni Experience 
	Builder.

targets -> Especifica los tipos de páginas Lightning a las que se puede agregar el componente 
	en Lightning App Builder.

targetConfigs -> permite especificar comportamiento específico para cada tipo de página 
	Lightning, incluidas cosas como los objetos que respaldan el componente.

Trabajar con datos ---------------------------------------

°°°°°°°° Usando lightning data service °°°°°°°°°

Es la forma preferida (y más sencilla) de trabajar con datos de Salesforce.
Se usa JavaScript para conectar componentes a los datos de Salesforce.

Mantiene datos consistentes y actualizados entre múltiples componentes y clientes a lo largo 
del ciclo de vida de la aplicación. Si varios componentes utilizan Lightning Data Service para 
trabajar con un registro, y uno de ellos actualiza el registro, los demás componentes reflejan 
automáticamente esa actualización.

Es la forma recomendada en Salesforce (LWC) para leer, crear, actualizar y eliminar registros sin
escribir Apex, usando datos “en caché” y respetando seguridad, permisos y FLS automáticamente.

||||||| Usando el componente form |||||||

<lightning-record-form
    *atributos*>
</lightning-record-form> -> Permite crear, ver o editar un registro.
	NO requiere JS obligatorio.
	Se usa cuando Necesitas un formulario rápido y No necesitas lógica compleja.
	No se debe usar cuando Quieres interceptar campos individuales, Necesitas UI muy 	
	personalizada y Necesitas validaciones personalizadas complejas.+
	Atributos: 
	- object-api-name="nombreObjeto" -> Define el objeto Salesforce.
		En Salesforce los objetos usan API Name en singular, no plural. ***
	- record-id
		- Si existe → modo edición o vista
		- Si NO existe → modo creación.
	- mode="valor" -> Define el comportamiento del formulario
		- view -> Solo lectura
		- edit -> Edición
		- readonly -> Similar a view
		- create -> Crear nuevo registro
	- layout-type -> Usa layouts de Salesforce.
		- Full -> Layout completo
		- Compact -> Compact Layout
	- columns="n" -> Número de columnas (solo en Full layout).
	- fields = ['nombreCampo1', 'nombreCampo2', 'nombreCampoN'];
	  fields={fields} -> sirve para decirle explícitamente al componente qué campos mostrar,
		y en que orden en lugar de usar el layout del objeto.
	Eventos disponibles: 
	- onsubmit={nombreFuncion} -> Funcion que se manda a llamar al presionar el button del 
		formulario.
		Solo se usa cuando Debes modificar campos antes de guardar, Debes validar datos 
		custom y Debes cancelar el guardado bajo ciertas condiciones.
	- onsuccess={nombreFuncion} -> Funcion que se manda a llamar después completar 
		correctamente la función onsubmit.
	- onerror={nombreFuncion} -> Funcion que se manda a llamar después la función onsubmit
		fallara.

||||||| Usando UI Record API en modo imperativo |||||||

Es un conjunto de funciones JS que te permite: Crear, Actualizar y Eliminar (y obtener info de 
metadata).
Se llama imperativo porque tú decides cuándo ejecutar la acción (click, evento, lógica).

Usa UI Record API cuando: Necesitas botones personalizados, Quieres lógica antes/después del
guardado y No quieres depender de layouts.

Ejemplo de guardar un registro:

handleCreate() {
    const fields = {
        Name: 'Cuenta creada por LWC',
        Phone: '555-1234',
        Industry: 'Technology'
    };

    const recordInput = {
        apiName: 'Account',
        fields
    };

    createRecord(recordInput)
        .then(record => {
            console.log('ID creado:', record.id);
        })
        .catch(error => {
            console.error(error);
        });
}


Ejemplo de actualizar un registro:

handleUpdate() {
    const fields = {
        Id: this.recordId,      // OBLIGATORIO
        Phone: '555-9999'
    };

    updateRecord({ fields })
        .then(() => {
            console.log('Registro actualizado');
        })
        .catch(error => {
            console.error(error);
        });
}

Ejemplo de eliminación de registro:

handleDelete() {
    deleteRecord(this.recordId)
        .then(() => {
            console.log('Registro eliminado');
        })
        .catch(error => {
            console.error(error);
        });
}


°°°°°°°° Usando apex °°°°°°°°°

Cuando quieres personalizar una transacción de datos de un solo registro, o realizar operaciones
multi-registro en una sola transacción, Apex es la mejor opción.


||||||| Usar métodos ápex |||||||

La anotación pone el método Apex a disposición de los componentes de Lightning (tanto los componentes 
web lightning como los de Aura).

*En apex*
public with sharing class nombreClaseController {} -> Se crea una clase de manera especial: la 	
parte de with sharing es para Respeta las reglas de sharing del usuario (Solo devuelve registos que
el usuario tiene permiso de ver).

*En apex*
@AuraEnabled(cacheable=true)
public static tipoDato nombreMetodoApex(parametros) {
    // una operación hacia la db
} -> La notación @AuraEnabled Permite que el método sea llamado desde: Lightning Web Components y
	Componentes Aura.
	- cacheable=true -> Indica que: El método NO modifica datos y Salesforce puede cachear 
		el resultado
		Cuando un método es cacheable, las versiones nuevas o modificadas de los 			
		registros no pueden devolverse hasta que la caché se actualice.

*xml*
<?xml version="1.0" encoding="UTF-8"?>
<LightningComponentBundle xmlns="http://soap.sforce.com/2006/04/metadata">
    <apiVersion>64.0</apiVersion>
    <isExposed>true</isExposed>
    <targets>
        <target>lightning__AppPage</target>
    </targets>
</LightningComponentBundle> -> Al configurar el XML se define dónde y cómo puede usarse tu 
	Lightning Web Component dentro de Salesforce.
	- <isExposed>true</isExposed> -> Hace que el componente sea visible en el Lightning App 
		Builder
		Si fuera false: No podrías arrastrarlo a ninguna página.
	- <targets> -> Define en qué tipos de páginas puede usarse el componente.
		- lightning__AppPage -> Indica que el componente puede colocarse en: App Pages.		
			NO podría usarse en: Record Pages y Home Pages


Hay dos formas de interactuar con los métodos Apex de los componentes de Lightning: o bien 
cablear el método o llamarlo imperativamente. 

Utilizando cables:

Forma para metodos: 

*En js*

Forma para mostrar una tabla en la vista: 

*En js*

import NOMBRE_CAMPO1_FIELD from '@salesforce/schema/NombreTabla.nombreCampo1';
import NOMBRE_CAMPO2_FIELD from '@salesforce/schema/NombreTabla.nombreCampo2';
import NOMBRE_CAMPON_FIELD from '@salesforce/schema/NombreTabla.nombreCampoN';
import nombreMetodoApex from '@salesforce/apex/NombreClaseController.nombreMetodoApex'; -> Nos 
	traemos todos los campos de una tabla que deseamos meter en la tabla e importamos la 
	función que nos trae los datos de ápex.

const COLUMNS = [
    { label: 'nombre columna 1', fieldName: NOMBRE_CAMPO1_FIELD.fieldApiName, type: 'tipoCampo' },
    { label: 'nombre columna 2', fieldName: NOMBRE_CAMPO2_FIELD.fieldApiName, type: 'tipoCampo' },
    { label: 'nombre columna N', fieldName: NOMBRE_CAMPON_FIELD.fieldApiName, type: 'tipoCampo' }
]; -> Para crear los metadatos de las columnas de las tablas.
	tipos de campo:
	- text
	- currency

columns = COLUMNS;
@wire(nombreMetodoApex)
nombreArregloDeRegistros; -> Para crear la tabla con los registros.

*En html*

<template if:true={nombreArregloDeRegistros.data}>
    <lightning-datatable
        key-field="Id"
        data={nombreArregloDeRegistros.data}
        columns={columns}
    >
    </lightning-datatable>
</template> -> Si hay registros en la tabla, los mostrara en una tabla.
	El componente lightning-datatable es componente que ya viene con Salesforce y se encarga
	de renderizar una tabla en la vista con los datos que le pasemos. Solo funciona en 
	escritorio y no en móvil. Para móvil se necesita crear un componente personalizado.


Utilizando forma imperativa:

Llama a Apex imperativamente cuando necesites controlar la invocación de las operaciones de 
lectura y cuando modifiques registros. 

*En js*
import { LightningElement, api, wire } from 'lwc';
importar nombreMetodoApex desde '@salesforce/apex/ContactController.nombreMetodoApex';
export default class nombreComponente extends LightningElement {
    @api nombreParameto;
    nombreFuncionEvento() {
        nombreMetodoApex({ 
            nombreParameto: this.nombreParameto
        })
        .then(records => {
            // Codigo si se presenta el error
        })
        .catch(error => {
            // Codigo si hay error
        });
    }
}
