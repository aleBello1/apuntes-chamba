Patrón circuit breaker -----------------------------------------------

Es un patrón que:
Detecta que las llamadas a Redis están fallando.
Deja de intentar llamadas inútiles.
Se recupera cuando Redis vuelve.

Se usa cuando se depende de un sistema externo que puede fallar o volverse lento. *****
Para el caso de redis, el punto es poder manejar el caso en el que redis no este funcionado y 
podamos usar la db (MySQL, postresql, etc) en su lugar.

Si no se implementa, las request serán lentas porque siempre se intentara llamar a Redis y como
este podría tardar en responder entonces habrá timeouts largos (si se configura el timeout aun
así será muy lento la request).

Estados del circuit breaker:
- closed: Significa que todo funciona correctamente (redis funcionando).
- open: Significa que hay errores (redis esta fallando) y en este caso se deja de llamar a redis.
- half-open: prueba de recuperación. 

Resilience4j: Es una librería de resiliencia para Java/Kotlin. Se le llama resiliente porque 
resiste fallos externos sin caerse. Implementa patrones de diseño para manejar fallos externos 
de forma inteligente.
Proporciona: Circuit Breaker, Retry, Rate Limiter, Bulkhead, Time limiter.
Resuelve problemas reales de producción: servicios lentos, dependencias caídas, cascadas de fallos,
saturación de threads.

Para aplicaciones con redis remoto se recomienda:

mínimo aceptable:
timeout corto
try/catch

Producción seria: 
timeout
Circuit Breaker + fallback
métricas + monitoreo (De este modo con aplicaciones de monitoreo sabremos el estado del circuito)

<dependency>
    <groupId>io.github.resilience4j</groupId>
    <artifactId>resilience4j-spring-boot3</artifactId>
</dependency> -> Dependencia para la resilience4J.
	Usado para spring boot v3.

resilience4j:
  circuitbreaker:
    instances:
      redisRemoteCB:
        slidingWindowSize: 10
        failureRateThreshold: 50
        waitDurationInOpenState: 30s
        permittedNumberOfCallsInHalfOpenState: 3 -> Configuración de resilience4J para el 
		circuit breaker.
		- instances:
      			nombreInstancia: -> Crea una instancia de configuración.
				Este nombre debe coincidir con el name de la anotación circuit
				breaker. 
				Si tuviéramos dos redis: uno local y otro remoto, se debería 
				crear una instancia de configuración para cada redis.
		- slidingWindowSize: n -> Analiza n llamadas.
			Por defecto es una ventana basada en número de llamadas.
			Pero puede configurarse como ventana basada en tiempo.
		- failureRateThreshold: m -> si el m% de las llamadas fallan → abre circuito.
			El valor m es un numero entre 1 y 100.
		- waitDurationInOpenState: ps -> Configura Cuánto tiempo el circuito permanece 
			ABIERTO antes de intentar probar otra vez.
			Cuando el circuito esta abierto NO se intentan llamadas a Redis, todo
			se va directo al fallback.
			Después de este tiempo pasa a half open (prueba si redis volvió).
			Si este valor es muy bajo la app seguirá intentando conectarse a cada
			rato y será una saturación innecesaria y si lo pones muy alto puede que
			redis este de regreso pero te quedes mucho tiempo sin usarlo.
		- permittedNumberOfCallsInHalfOpenState: q -> Cuando el circuito pasa a 
			HALF-OPEN Solo se permiten q llamadas reales para probar si el sistema 
			ya se recuperó.
			Si el porcentaje de fallos en HALF-OPEN supera el threshold → vuelve a 
			OPEN.

@CircuitBreaker(name = "redisRemoteCB", fallbackMethod = "fallback")
fun <T> get(key: String, typeRef: TypeReference<T>): T? {
    val json = stringRedisTemplate.opsForValue().get(key)
        ?: return null

    return objectMapper.readValue(json, typeRef)
} -> Método que se usa para recuperar datos de redis si es que la clave existe.
	Si el método falla (por ejemplo porque no hay conexión con redis porque esta caído
	o si simplmenete se supera el timeout), se lanzara una excepción y ejecutara el método
	que se asigne al parámetro: 'fallbackMethod'.
	- fallbackMethod = "fallback" -> El método fallback debe cumplir reglas estrictas: Debe
 		estar en la misma clase, Debe tener los mismos parámetros que el método que se 
		le pone la anotación CircuitBreaker y Debe agregar un parámetro extra al final
		de tipo Throwable.
		Cuando el circuito este abierto, el método con la anotación CircuitBreaker ni 	
		siquiera se ejecuta e inmediatamente se ejecuta su fallback.
	- name = "redisRemoteCB" -> Identifica una instancia específica del Circuit Breaker (de 
		los que se configuraron en el application en la opción instances: 		
		nombreInstancia).
		La conecta con una configuración concreta.
		Mantiene su propio estado independiente.

import com.fasterxml.jackson.core.type.TypeReference // Tiene que venir de Jackson.

fun <T> fallback(
    key: String,
    typeRef: TypeReference<T>,
    ex: Throwable
): T? {
    println("No se encontro esta request en redis: ${ex.message}")
    return null
} -> Método que se ejecuta si el método get falla.
	Si por alguna razón dentro de este método se pone un código que pueda lanzar una 
	excepción, esa ya no lo intercepta el circuit breaker y esa excepción si se propaga hacia
	arriba.

@CircuitBreaker(name = "redisRemoteCB", fallbackMethod = "fallbackSet")
fun set(key: String, value: Any, ttlHours: Long) {
    val json = objectMapper.writeValueAsString(value)
    stringRedisTemplate.opsForValue()
        .set(key, json, ttlHours, TimeUnit.HOURS)
} -> Método que se usa para insertar nuevos datos a redis con determinada clave.

fun fallbackSet(key: String, value: Any, ttlHours: Long, ex: Throwable) {
    println("No se pudo guardar esta request en redis: ${ex.message}")
} ->  Método que se ejecuta si el método set falla.

fun getNombreRecurso(): List<tipoDatoX> {

    val key = "nombreClave"

    val cached = redisCacheRepository.get(
        key,
        object : TypeReference<List<tipoDatoX>>() {}
    )

    if (cached != null) return cached

    val fromDb = nombreEntidadService.getNombreRecursoFromDb()

    redisCacheRepository.set(key, fromDb, tiempo)

    return fromDb
} -> Método para aplicar Cache Aside (Lazy Loading).
	Busca cierto recurso en redis si existe lo retornar pero si no lo busca en la db, lo 
	guarda en redis le aplica un ttl y lo retorna.
	Estos métodos a serian los que se mandarían a llamar en los controllers.

Los que se implementa funciona para Ver cuánto tiempo el Circuit Breaker estuvo OPEN, Ver histórico

Micrometer es una librería de métricas para JVM. Es la capa intermedia entre tu aplicación y el 
sistema de monitoreo.
Se encarga de Micrometer de Crear métricas (counter, gauge, timer), Las registra y Las exporta 
al sistema que tú elijas.
Desde Spring Boot 2, Micrometer viene integrado automáticamente.

Prometheus es un sistema de monitoreo basado en scraping.
Encargado de:
- Hace requests HTTP a tu app
- Lee métricas en formato especial
- Las guarda en una base de datos de series temporales (Prometheus tiene su propia base de datos de 
series temporales (TSDB), No vuelve lenta tu app, sin embargo tener una cardinalidad alta
puede ser peligroso porque puede matar al servidor de prometheus).
- Permite hacer queries y alertas
Prometheus NO recibe métricas.
Él las consulta periódicamente.
De lo mas importante es que indica si la app esta viva, cuanta cpu usa y cuanta memoria.

Grafana es una herramienta de visualización de métricas. NO guarda métricas y no recoleta 
métricas.
Encargado de:
- conecta a una fuente de datos (ej: Prometheus)
- Hace consultas
- Construye dashboards
- Muestra gráficas
- Permite configurar alertas **

Spring Boot usa:
Micrometer → para generar métricas.
Spring Boot Actuator → para exponerlas por HTTP.
Prometheus → para recolectarlas.
Grafana permite visualizarlas.

CONFIGURAR SPRING BOOT PARA QUE GUARDE LOS LOGS EN UN ARCHIVO

Logback: sistema de logging por defecto en Spring Boot.

rollover: es el proceso mediante el cual un sistema de logs archiva o renombra un archivo de
log actual cuando alcanza un cierto tamaño o tiempo, y crea un nuevo archivo para seguir 
registrando los logs.

Niveles de log (de menor a mayor gravedad):
Nivel	Descripción
TRACE	Muy detallado. Se usa para depurar paso a paso el flujo del programa.
DEBUG	Información útil para desarrolladores al depurar.
INFO	Información general sobre el estado de la aplicación.
WARN	Indica algo inesperado, pero la aplicación aún puede continuar.
ERROR	Indica un problema que ha causado una falla en la aplicación o función.
OFF	Desactiva completamente los logs.

Ejems:

log.trace("Inicializando servicio...");
log.debug("Parámetros recibidos: {}", params);
log.info("Servicio iniciado correctamente");
log.warn("Tiempo de respuesta alto");
log.error("No se pudo conectar a la base de datos");

*En aplication.properties*

logging.file.max-history=num -> Número máximo de archivos que se guardan.

logging.level.root=NIVEL_LOG -> Configura que nivel de log se registrara.
	Se regristraran los logs de toda la app. ****
	Nivel de log: TRACE, DEBUG, INFO, WARN ó ERROR.
	Si se usa el nivel de log INFO, solo veremos los logs de INFO, WARN y ERROR mientras
	que los logs de TRACE y DEBUG serán ignorados, porque están por debajo del nivel 
	INFO. ***

Configuración de logs usando un archivo xml ---------------------

En src/main/resources se crea un archivo llamado logback-spring.xml. Spring Boot ya trae una 
configuración interna por defecto.
Si tú creas logback-spring.xml, la reemplazas completamente.

Se puede usar configuración en application.yml o logback-spring.xml.
Cuando se define logback-spring.xml, normalmente se deja de usar la configuración básica de 
logging en application.

<?xml version="1.0" encoding="UTF-8"?>
<configuration>
	…
</configuration> -> Para configurar todo lo relacionado a los logs, como propiedades, políticas
	de rotación y niveles de logs.
	Es el nodo raíz obligatorio del archivo de configuración de Logback.

<include resource="rutaArchivo"/> -> Permite importar configuraciones de Logback definidas 
	en otro archivo XML.
	Spring Boot normalmente incluye: org/springframework/boot/logging/logback/base.xml
	Este archivo ya define las configuraciones de la consola. En resumen agragar esta 
	nos permite tener la impresión de logs en la consola como nomalmente se tendría.

<springProfile name="nombrePerfil">
    *Etiqueta de configuracion*
</springProfile> -> Para que cierta etiqueta de configuración funcione solo cuando se tenga
	activo determinado perfil.

<property name="NOMBRE_VARIABLE" value="valor"/> -> Define una variable reutilizable.
	Es común que el nombre la variable este escrita en mayúsculas.
	Para utilizar esta variable en otro lado se usa asi: ${NOMBRE_VARIABLE}.
	Las propiedades pueden ser reutilizadas en cualquier parte del archivo después de ser 
	declaradas.

<appender name="nombre" class="tipoAppender">
	… 
</appender> -> Indica el destino donde se escribiran los logs.
	Atributos:
	- name="nombre" -> Esto es solo un identificador.
		No tiene un significado especial, sin embargo, el nombre que se ocupe aquí debe
		ser único y coincidir con el que se coloque aquí: <appender-ref ref="..."/>.
	- class="tipoAppender" -> Define el tipo de appender.
		- ch.qos.logback.core.rolling.RollingFileAppender -> Para indicar que se 
			Escribira en archivo y puede rotar automáticamente.
		- ch.qos.logback.core.ConsoleAppender -> Para escribir en la terminal.
		- ch.qos.logback.core.FileAppender -> Escribe en archivo PERO sin rotación.
			No recomendado para producción porque el archivo crecerá infinito.
	Configuraciones dentro de esta etiqueta: 
	En caso de que se escriba en un file ----------------------
	- <file>"rutaX/nombreARchivo.log"</file> -> Define la ruta y nombre del archivo.
	- <rollingPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedRollingPolicy">
		...
	  </rollingPolicy> -> Define la o las reglas que controlan cuándo se crea un nuevo 
		archivo.
		Configuraciones dentro de esta etiqueta: 
		- <fileNamePattern>...</fileNamePattern> -> Define cómo se llamarán los archivos
			rotados.
			El %d indica que se comenzara a escribir el formato de la fecha y el %i
			representa el índice que se incrementará cuando el archivo supere el
			maxFileSize.
		- <maxFileSize>tamanio_y_unidad</maxFileSize> -> Si en un mismo día el archivo 
			supera este tamaño se cierra y se crea uno nuevo con el índice + 1.
			Esto para evitar archivos gigantes.
		- <maxHistory>n</maxHistory> -> Mantiene máximo n días de logs.
			El dia n + 1 automáticamente eliminará el mas antiguo.
		- <totalSizeCap>tamanio_y_unidad</totalSizeCap> -> Define el Límite total 
			acumulado de todos los archivos rotados si la suma supera este limite:
			Empieza a borrar los más antiguos.
			Este límite se aplica a los archivos rotados, no al archivo activo 
			actual.
	Se ocupa para file o console ----------------------
	- <encoder>
    		<pattern>*formato log*</pattern>
	  </encoder> -> Define el formato del log.
		Ejemplo de formato: %d{yyyy-MM-dd HH:mm:ss} %-5level %logger - %msg%n 
		- %d{yyyy-MM-dd HH:mm:ss} → fecha y hora
		- %-5level → nivel (INFO, WARN, ERROR)
		- %logger → clase que genera el log
		- %msg → mensaje
		- %n → salto de línea
	
<root>
    <appender-ref ref="nombre"/>
</root> -> Aquí se meten el nombre de todos los appender configurads para poder registrar los 
	logs en determinados destinos y con determinadas reglas.
	- level="nombreNivel" -> sirve Para indicar los niveles que deben tener los logs 
		para ser registrados, el nivel que se coloque indicara que se deben registrar 
		los logs con ese nivel y niveles mas con mayor gravedad.
		Si el root está en INFO, entonces se registran INFO, WARN y ERROR.
	- name="rutaPaquete" -> Permite configurar niveles diferentes para paquetes específicos.

CARPETAS DE LINUX

/opt: se usa principalmente para instalar software adicional o de terceros que no forma parte del sistema base.
Esa carpeta es como un lugar “aislado” donde cada aplicación puede vivir con todas sus 
dependencias sin afectar al resto del sistema.
Ejemplos típicos:
/opt/google/chrome
/opt/intellij-idea
/opt/postman

/usr: Se utiliza para instalar software del sistema (gestionado por el sistema).

*debajo de control de flujo*
	Logicos:
	corchetes simples:
	Strings ----------------------------------------
	- -z cadena -> Pregunta si la cadena esta vacía.
	- -n cadena -> Pregunta si la NO cadena esta vacía.
	Archivos ---------------------------------------
	- -f nombreArchivo.extension -> Pregunta de la existencia de un archivo.
	- -d nombreCarpeta -> Pregunta de la existencia de una carpeta.
	- -e nombreArchivo.extension ó nombreCarpeta -> Pregunta de la existencia de una carpeta o archivo.




GESTION DE SERVICIOS EN LINUX CON SYSTEMD PARA LAS APPS

La siguiente teoría sirve para explicar como convertir un jar en un servicio administrado por
linux que le permita reiniciar la app cada que se cae y correrlo en segundo plano.

sudo mkdir -p /opt/jar-service -> Crea el lugar donde vive el controlador, es decir, el script que controlara
	al archivo .jar.

sudo mkdir -p /home/ubuntu/app
sudo chown -R ubuntu:ubuntu /home/ubuntu/app
sudo chmod 755 /home/ubuntu/app -> Para Asegurar que el directorio de la app existe y pertenece al usuario.

sudo chown ubuntu:ubuntu /opt/jar-service
sudo chmod 755 /opt/jar-service -> Darle permisos para el directorio del script.

sudo nano /opt/jar-service/jar-ws.sh -> Creación y edición del script.

sudo chmod +x /opt/jar-service/jar-ws.sh -> Darle permisos de ejecución al script.


Instrucciones dentro del archivo ----------------------------------------------------------

#!/bin/bash

APP_DIR="/home/ubuntu/app" -> Define las rutas donde viven los jars.

JAVA_BIN=$(which java || echo "/usr/bin/java") -> Intenta encontrar Java en el sistema.
	- which java → Busca en el PATH el comando llamado java.
	- si falla → usa /usr/bin/java (Evita errores si PATH no está bien configurado).

JAVA_OPTS="-Xms256m -Xmx512m -XX:+UseG1GC" -> Configuración la memoria de la app.
	- -Xms256m → Define la memoria mínima que usara la app.
	- -Xmx512m → Define la memoria maxima que usara la app.
	- -XX:+UseG1GC → Para usar el garbage collector moderno.

JAR_FILE=$(ls -t "$APP_DIR"/*.jar 2>/dev/null | head -n 1) -> Para buscar y tomar el jar mas 
	reciente.
	- ls -t /home/ubuntu/app/*.jar -> Lista los .jar ordenados por fecha de modificación: El 
		más reciente → arriba y El más viejo → abajo.
	- 2>/dev/null -> Oculta errores (por ejemplo si no hay .jar).
	- | head -n 1 -> Toma solo el primer elemento.

if [ -z "$JAR_FILE" ]; then
    echo "ERROR: No se encontró ningún archivo .jar en $APP_DIR"
    exit 1
fi -> Si no se encontró ningún jar imprime un mensaje y termina el script (systemd lo verá como 
	fallo).

cd "$APP_DIR" || exit -> Nos movemos a donde esta la app pero si falla termina el script.

echo "Iniciando Spring Boot..."
echo "JAR: $JAR_FILE"
exec "$JAVA_BIN" $JAVA_OPTS \
    -jar "$JAR_FILE" \
    --server.port=5000 \
    --spring.profiles.active=nombreAmbiente -> Para levantar el jar utilizando ciertas opciones.


Configuración del servicio -------------------------------------------------------------

systemd: Es el sistema que controla TODO lo que corre en Linux. 
	Rutas:
	- /etc/systemd/system -> 

sudo nano /etc/systemd/system/jar-ws.service
	El servicio se crea dentro de esta ruta porque en esta ruta van todos los serivicos personalizados
	del usuario y admin.

Secciones del archivo de configuración:
[Unit]     → qué es y cuándo iniciar
[Service]  → cómo se ejecuta
[Install]  → cuándo se activa automáticamente

°°°°°°°°° Instrucciones dentro de unit °°°°°°°°

Description=texto -> Agrega una description meramente para humanos
	Esta se visualizara al hacer systemctl status nombreServicio.

After=network.target -> Para que Linux arranque el servicio después de que otro servicio este listo.
	Ejemplos:
	- After=network.target -> Para que Linux arranque el servicio después de que la red esté 
		lista.





Restart=always -> Si la app se cae se levanta sola.

User=Ubuntu -> Lo corre con determinado usuario.

MemoryMax=800M -> Limita consumo de RAM.

Ahora systemd captura los logs de la app y podemos utilizar el siguiente comando.

journalctl -u jar-ws -> Para mostrar los logs de determinado servicio.
	journald es el sistema de logs de Linux.
	Los logs normalmente se guardan en disco en /var/log/journal/.
	Journalctl usa RAM como buffer temporal: Para escribir rápido y para no saturar el disco con cada 
	log después los pasa al disco.
	Al hacerlo asi, no importa donde este el archivo, siempre podremos pedir los logs.
	- journalctl -u jar-ws -f -> Para ver los logs den tiempo real.
