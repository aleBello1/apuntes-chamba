METRICAS 

outliers: Son valores anormalmente altos o bajos comparados con el resto.
	Ejemplo en la app si hay 99 requests → 100 ms y 1 request → 5 segundos, entonces ese 
	de 5s es un outlier. ****


Spring boot app -------------------------------

http_server_requests_seconds_count -> Es un counter. Contiene el numero total acumulado de 
	requests HTTP que tu aplicación ha recibido.
	Empieza en 0 cuando arranca la app y nunca disminuye.
	Cuando la app se reinicia, el counter vuelve a 0, Prometheus detecta ese reset 
	automáticamente
	Parámetros/labels que trae:
	- method -> Método HTTP (GET, POST, etc.)
	- status -> Codigo de status http de la request (200, 404, 500, etc)
	- uri -> Url del servicio
	- outcome -> Nombre del codigo http (SUCCESS, CLIENT_ERROR, SERVER_ERROR)
	- exception -> Nombre de la excepción si ocurrió error (puede estar vacío).

http_server_requests_seconds_sum -> tiempo TOTAL acumulado de todas las requests.
	Ejemplo: Request 1 → 1s; Request 2 → 2s; Request 3 → 3s entonces sum = 6 segundos.
	
jvm_memory_used_bytes{area="heap"} -> Muestra la memoria HEAP usada por la JVM, separada por
	regiones (id).
	GC: Garbage Collector (Recolector de basura). Es el sistema automático de Java que:
	libera memoria eliminando objetos que ya no se usan. 
	Regiones: 
	- Eden -> Memoria usada por objetos recién creados.
		Se llena y se limpia constantemente (GC joven)
	- Survivor -> Memoria usada por Objetos que sobrevivieron a un GC.
		Zona de transición hacia Old Gen
	- Old gen -> Memoria usada por objetos viejos.
	- sum(jvm_memory_used_bytes{area="heap"}) -> Representa la memoria total de ram que la 
		aplicación está usando en este momento. ******
		Si la app llega al limite se muere.

sum(jvm_memory_max_bytes{area="heap"}) -> Representa el límite máximo que la JVM puede usar 
	para heap.

process_cpu_usage -> Porcentaje de CPU que está usando la app Java.
	
resilience4j_circuitbreaker_state -> Contiene el estado del circuit breaker.

resilience4j_circuitbreaker_calls_total -> Es un contador que dice cuántas llamadas han 
	pasado por el circuit breaker, clasificadas por resultado.
	Parámetros/labels que trae:
	- kind="successful" -> Llamadas que pasaron y salieron bien.
	- kind="failed" -> Llamadas que pasaron pero fallaron
	- kind="not_permitted" -> Llamadas que fueron BLOQUEADAS.
		significa el circuit breaker está OPEN.

Node exporter -----------------------------------------

Se tendría que levantar con Docker compose un node exporter en la instancia que se desea
monitorear.

node_memory_MemTotal_bytes -> Contiene toda la RAM física de tu servidor.
	Incluye todo: lo usado + libre + cache.

node_memory_MemAvailable_bytes -> Contiene la RAM realmente disponible para usar sin problemas.

node_cpu_seconds_total -> Porcentaje de CPU que está usando el servidor.
	Es una métrica acumulativa por eso SIEMPRE se usa algo como: 
	rate(node_cpu_seconds_total[1m])

node_filesystem_size_bytes → tamaño en disco total

node_filesystem_avail_bytes → espacio en disco disponible

CONSULTAS PROMQL

Promql significa lenguaje de consultas de prometheus.

nombreMetricaZ{nombreParametroX operador valorX} -> Para condicionar que la metrica deba tener 
	cierto valor en determinado parámetro.
	-  nombreParametro1 operador valor1, nombreParametro2 operador valor2, nombreParametroN 
		operador valorN -> Se colocan comas para condicionar por varios labels, la coma
			seria equivalente a un and.
	Ejemplo: {status=~"4.."} seria para seleccionar todos los datos que en el parámetro 
		status inicie con 4xx.
	Operadores: 
	- != → Diferente
	- =~ -> Para indicar que lo siguiente será una expresión regular.
		Operadores usados con regex:
		- | -> Para indicar un valor u otro. Se puede usar multiples veces.

nombreMetricaZ[tiempo] -> Para aplicar un range vector.
	Significa Toma los valores de esta métrica en los últimos n unidad de tiempo.
	Ejemplo: 5m seria los últimos 5 minutos.
	Unidades:
	- s	segundos
	- m	minutos
	- h	horas
	- d	días
	- w	semanas
	- y	años

Funciones -----------------------------

rate(...) -> Calcula la velocidad promedio por segundo a la que está creciendo un contador.
	Devuelve un valor por segundo.
	Se usa principalmente para métricas tipo Counter.
	Se usa mucho para dashboards mientras que para alertas se ocupa increse. ************

by() -> Permite hacer un group by.

increase(metrica[tiempo]) -> Calcula cuánto aumentó el contador dentro del rango de tiempo que 
	le pasas. **
	Solo tiene sentido en metricas de tipo Counters.
	Ejemplo si increase(contador[5m]) y si el primer valor es a las 10.00am y hubo 30 	
	mientras que a las 10.05am hubo 47 entonces se graficara a las 10.00am un 30 y a las 
	10.05am se graficara un 17.


CONFIGURACION Y LEVANTAMIENTO DE PROMETHEUS

En la raíz del proyecto se crea el archivo de configuración de prometheus llamado prometheus.yml.
 
global: -> Para hacer configuraciones globales de prometheus.
	Configuraciones para esta opción:
  	- scrape_interval: ns -> Cada n segundos Prometheus hará una petición HTTP a las apps, db's
		o herramientas para leer métricas.
		El valor 15 es el recomendado para producción.
	- scrape_timeout: xs -> define cuánto tiempo espera Prometheus a que un target responda antes 
		de cancelar el scrape.
		Cada n segundos Prometheus intenta leer métricas, Pero solo espera máximo x segundos 
		a que respondan.
		Si no existiera timeout Prometheus podría quedarse esperando Por eso la regla 
		recomendada es: scrape_timeout < scrape_interval
		El valor 10 es el recomendado para producción.
	- evaluation_interval: zs -> Cada z segundos Prometheus evalúa reglas.
		El valor 15 es el recomendado para producción.
		Solo se configura si no se usan las alertas de grafana.

scrape_configs: -> Lista de cosas que Prometheus debe monitorear.
	Se coloca una lista de Jobs, cada elemento nuevo empieza por el carácter '-'.
	- job_name: "prometheus"
            static_configs:
              - targets: [ "localhost:9090" ]
                labels:
                  service: "prometheus" -> Para exponer métricas internas de prometheus.
  	- - job_name: "nombreJob" -> Asigna que monitoreo determinado job.
		Un job puede ser una app, una herramienta o una db.
	- metrics_path: "ruta" -> Le dice a Prometheus la URL donde debe leer 
		la métrica para este job.
		- /actuator/prometheus -> Para leer métricas de la app.
	- metric_relabel_configs: -> Define reglas que permiten modificar o eliminar métricas antes 
		de que Prometheus las almacene.
	    - - source_labels: [__name__]
              regex: "promhttp_.*"
    	      action: drop -> Muchos sistemas de observabilidad están escritos en Go. Por ejemplo: 
		Prometheus, Grafana Loki, Grafana Tempo y OpenTelemetry Collector. Todos generan
		métricas internas del runtime de Go.
		Como prometheus recibe muchas métricas del runtime de go y estas métricas no sirven para 
		dashboards lo mejor es eliminarlas, asi se reduce la ram, disco y cardinalidad.
		Esta configuración sirve para eliminar todas las métricas de go. Esta configuración se 
		usa sobre collector, tempo y loki.
	Opciones si se dará de alta un target por su ip ---------------------------------
	- static_configs:
		- - targets: ["ipApp:puertoApp"] -> Indicarle a que servidor conectarse, es decir, 
			indicarla donde esta la app, db o herramienta que debe consumir para recolectar 
			las métricas.
			Si se se hace la suposición que la app esta corriendo en local y prometheus
			dentro de un contenedor, evidentemente el locahost será: 'host.docker.internal'.
		- labels:
  		    service: "nombreLabel" -> Agrega un label a todas las métricas provenientes
			de este target (herramienta, db o app).
			Esto ayuda en consultas por eso es importante agregarlo y que sea el 
			nombre del target.
	Opciones si se dará de alta un target por su dominio ---------------------------------
	- scheme: https -> Indica qué protocolo usar para hacer el scrape.
		Otro posible valor seria http esto dependerá del dominio.
	- static_configs:
		- - targets: ["dominio"] -> Indicarle a que servidor conectarse, es decir, 
			indicarla donde esta la app, db o herramienta que debe consumir para recolectar 
			las métricas.
			Se coloca el puro dominio sin el 'https' ni las diagonales y ni los dos 
			puntos. ****
		- labels:
  		    service: "nombreLabel" -> Funciona igual, va el nombre de la app.

Hacer que promethus consuma las métricas: 

docker network create red -> Se crea una red para que compartan prometheus y grafana.

MSYS_NO_PATHCONV=1 docker run -d \
  --name prometheus \
  -p 9090:9090 \
  -v ${PWD}\prometheus.yml:/etc/prometheus/prometheus.yml \
  -v prometheusData:/prometheus \
  --network red \
  --memory=2g \
  --restart unless-stopped \
  prom/prometheus \
  --storage.tsdb.retention.time=15d \
  --storage.tsdb.wal-compression \
  --storage.tsdb.path=/prometheus \
  --query.max-concurrency=20 \
  --storage.tsdb.retention.size=2GB \
  --web.enable-lifecycle \
  --query.timeout=2m -> Crear un contenedor con base a la imagen prom/prometheus.
	Banderas de docker --------------------------
	- -p 9090:9090 -> Puerto donde queda expuesto prometheus.
	- -v $(pwd)/prometheus.yml:/etc/prometheus/prometheus.yml -> Es un volumen al cual se le
		dice Usa MI prometheus.yml personalizado En lugar del default del contenedor.
	- --memory=2g -> el contenedor no podrá usar más de 2 GB de RAM.
	- --restart unless-stopped -> Si Docker mata el contedor entonces se reinicia 
		automáticamente.
	Banderas de prometheus --------------------------
	Las banderas de prometheus deben ir después de la imagen porque son banderas de la imagen y 
	no de Docker.
	- --storage.tsdb.path=/prometheus -> Define dónde se guardan las métricas en disco.
	- --storage.tsdb.retention.time=nd -> Para que promethues guarde métricas de máximo n días.
	- --storage.tsdb.retention.size=xGB -> Manera alternativa de retención. Prometheus mantiene 
		máximo xGB de métricas, cuando supera el limite empieza a borrar métricas antiguas.
	- --storage.tsdb.wal-compression -> Para comprimir el WAL y reducir el disco.
	- --web.enable-lifecycle -> Permite recargar la configuración ionicial sin reiniciar 
		Prometheus. De este modo Prometheus vuelve a leer: prometheus.yml
	- --query.max-concurrency=20 -> controla cuántas consultas simultáneas puede ejecutar Prometheus.
	- --query.timeout=2m -> si una query tarda más de 2 minutos la cancela

GET /targets -> En esta ruta prometheus enlista todas los targets que esta scrapenado/consumiendo, 
	ademas muestra de cada uno si están up ó down y cada cuanto las consume.
	curl http://localhost:9090/api/v1/targets

GET /query -> Muestra que métricas tenemos disponibles con base a los endpoints que esta 
	consumiendo prometheus.

POST /-/reload -> Para obligar a prometheus a volver a cargar su archivo de configuración. *******
	Entonces cada que queramos monitorear un nuevo proyecto tendremos que agregar los datos del proyecto
	al archivo de configuración, seguidamente usar este endpoint para que prometheus note los cambios
	del archivo de configuración.
	curl -X POST http://localhost:9090/-/reload
