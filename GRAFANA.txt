METRICAS 

Spring boot app -------------------------------

http_server_requests_seconds_count -> Es un counter. Contiene el numero total acumulado de 
	requests HTTP que tu aplicación ha recibido.
	Empieza en 0 cuando arranca la app y nunca disminuye.
	Cuando la app se reinicia, el counter vuelve a 0, Prometheus detecta ese reset 
	automáticamente
	Parámetros/labels que trae:
	- method -> Método HTTP (GET, POST, etc.)
	- status -> Codigo de status http de la request (200, 404, 500, etc)
	- uri -> Url del servicio
	- outcome -> Nombre del codigo http (SUCCESS, CLIENT_ERROR, SERVER_ERROR)
	- exception -> Nombre de la excepción si ocurrió error (puede estar vacío)
	

CONSULTAS PROMQL

Promql significa lenguaje de consultas de prometheus.

nombreMetricaZ{nombreParametroX operador valorX} -> Para condicionar que la metrica deba tener 
	cierto valor en determinado parámetro.
	-  nombreParametro1 operador valor1, nombreParametro2 operador valor2, nombreParametroN 
		operador valorN -> Se colocan comas para condicionar por varios labels, la coma
			seria equivalente a un and.
	Ejemplo: {status=~"4.."} seria para seleccionar todos los datos que en el parámetro 
		status inicie con 4xx.
	Operadores: 
	- != → Diferente
	- =~ -> Para indicar que lo siguiente será una expresión regular.

nombreMetricaZ[tiempo] -> Para aplicar un range vector.
	Significa Toma los valores de esta métrica en los últimos n unidad de tiempo.
	Ejemplo: 5m seria los últimos 5 minutos.
	Unidades:
	- s	segundos
	- m	minutos
	- h	horas
	- d	días
	- w	semanas
	- y	años

Funciones -----------------------------

rate(...) -> Calcula la velocidad promedio por segundo a la que está creciendo un contador.
	Devuelve un valor por segundo.
	Se usa principalmente para métricas tipo Counter.

by() -> Permite hacer un group by.

increase(metrica[tiempo]) -> Calcula cuánto aumentó el contador dentro del rango de tiempo que 
	le pasas. **
	Solo tiene sentido en metricas de tipo Counters.
	Ejemplo si increase(contador[5m]) y si el primer valor es a las 10.00am y hubo 30 	
	mientras que a las 10.05am hubo 47 entonces se graficara a las 10.00am un 30 y a las 
	10.05am se graficara un 17.

////////////////////////INSTRUMENTAR SPRING BOOT CON OPENTELEMETRY (LOCAL)//////////////////////

Instrumentar una aplicación significa: Agregarle la capacidad de medir y reportar lo que está haciendo 
internamente.
Si un servicio está instrumentado:
- Mide cuánto tardó.
- Mide cuántas veces se llamó.
- Mide si fue error 500.
- Guarda un trace con todos los pasos internos.
- Puede reportar métricas, logs y trazas.

OpenTelemetry es un estándar open source para instrumentar aplicaciones.
Puede enviar logs, métricas y traces.

Grafana tempo: Es la base de datos donde se guardan los requests de tu aplicación (las trazas).
Una traza es El recorrido completo de una petición dentro del sistema.
Span: Una operación individual dentro del trace

El flujo ideal de herramientas debe ser el siguiente:

App (Combinado con el agente OpenTelemetry)
        ↓
Collector (receiver → processor → exporter)
        ↓
Tempo (almacena trazas)
        ↓
Grafana (visualiza)


CONFIGURACION E IMPLEMENTACION DE TEMPO

Si solo se ocupa openTelemetry + tempo (sin el collector en medio de los dos), primero se debe crear
el contenedor de tempo porque openTelemetry se conectara a él.

En la raíz del proyecto se debe crear un archivo llamado tempo.yaml y tener ciertas configuraciones.

Las siguientes configuración solo sirven para tempo versión 2.7.1 (Para determinarlo nos metimos
al contenedor y ejecutamos tempo --version).

stream_over_http_enabled: true -> Permite que Tempo envíe resultados de consultas a Grafana usando 
	streaming HTTP.
	Al hacerlo con stream Tempo empieza a mandar datos mientras grafana todavía los está 
	leyendo. Esto provoca consultas más rápidas.

server:
	Configuraciones para esta opción:
	- http_listen_port: 3200 -> Abre un servidor HTTP en el puerto 3200.
		Ese puerto sirve para: Que Grafana se conecte a Tempo, Consultar trazas y Hacer 
		búsquedas.
		Este puerto NO es para que tu aplicación envíe trazas.
	- log_level: nivelLog
		- debug -> muy detallado
		- info	normal
		- warn	advertencias
		- error	errores

distributor: -> se encarga de recibir spans, validarlos y distribuirlos a los ingesters.
	- receivers: -> Define qué protocolos acepta Tempo para recibir trazas.
		- otlp: -> Para decir que se recibirá trazas usando el protocolo OTLP.
			Configuraciones para esta opción:
      			- protocols:
         			- grpc:
				    endpoint: 0.0.0.0:4317
         		 	  http: 
				    endpoint: 0.0.0.0:4318 -> Habilitar dos formas de recibir 
					trazas: via grpc (puerto 4317) y http (puerto 4318) en 
					todas las interfaces. Escuchar en todas las interfaces de red
					permite conexiones desde otros contenedores y otras máquinas.

ingester: -> Para configurar al intester.
	Los ingesters reciben las trazas desde el distributor y las preparan para almacenamiento.
	Configuraciones para esta opción:
	- trace_idle_period: ns -> Tempo espera n segundos sin recibir spans nuevos para 
		considerar que la traza está completa.
		Ejemplo: traceID 123 -> span1 -> span2 -> span3. Si en n segundos no llegan más 
		spans entonces Tempo cierra la traza.
	- max_block_bytes: 100 -> Tamaño máximo de bloque en memoria antes de escribirlo.
		Tempo no guarda cada span inmediatamente en disco.
		Primero los acumula en memoria y Cuando el bloque alcanza cierto tamaño o tiempo
		se guarda en el storage.
		En prod, el valor recomendado es 500MB (524288000 bytes) hasta 1GB y para pruebas
		se puede dejar entre 1MB a 10MB.
	- max_block_duration: 30m -> controla el tiempo máximo que un bloque puede permanecer 
		abierto.
		Este parámetro evita que un bloque permanezca abierto demasiado tiempo si el tráfico 
		es bajo.
		Entonces si el bloque no alcanza el max_block_bytes pero ya paso 
		max_block_duration el bloque se cierra.
		En prod el valor recomendado entre 30m y 1h.

storage: -> Define dónde se guardan las trazas.
  trace:
    backend: tipoUbicacion -> Indica el tipo de almacenado de las trazas.
	Ubicaciones:
	- local
	- s3
	- gcs
	- azure
	Configuraciones para esta opción:
	- wal:
	    path: rutaX -> Configura el wal (Write Ahead Log) y su ruta. Es un log de seguridad. Antes 
		de guardar datos en almacenamiento final tempo escribe en wal.
		Si tempo crashea se recupera desde wal.
	Si se uso local: -----------------------------
	- local: # se repite esta linea
	    path: /tmp/tempo/traces -> Indica la ruta en la maquina local donde se guardaran las
		archivos.
		Si tempo esta dentro de un contenedor y se quiere implementar un volumen de persistencia
		esta ruta debe coincidir con la ruta del volumen.
	
compactor: -> Optimiza los bloques almacenados.
  compaction:
	- block_retention: n[unidad_tiempo] -> Guarda trazas solo por n cantidad de tiempo después
		trazas que superen esta antigüedad será eliminado automáticamente.
		Si esto no se configura siempre se guardarán las trazas hasta que el disco se 
		llene.
		Unidades:
		- h	horas
		- m	minutos
		- s	segundos
	- compacted_block_retention: 1h -> Mantiene los bloques viejos un tiempo antes de 
		borrarlos.
		El valor en prod recomendado es entre 1h y 2h.

overrides:
  defaults: -> sirve para limitar cuántos datos podemos recibir de cada 
	cliente
	Configuraciones para esta opcon:
	- max_traces_per_user: x -> Define el número máximo de trazas activas que un usuario 
		puede tener en memoria.
		Si llega a x trazas activas, Tempo empieza a rechazar nuevas trazas.
    	- ingestion_rate_limit_bytes: y -> limita cuántos bytes por segundo puede enviar un 
		cliente.
		si lo supera Tempo empieza a responder: 429 Too Many Requests.

query_frontend:
  search:
    duration_slo: ns -> componente que optimiza consultas desde Grafana.
	divide consultas grandes, paraleliza búsquedas y cachea resultados.
	Para indicar que Las búsquedas deberían resolverse en máximo n segundos.

metrics_generator:
  registry:
    external_labels:
      source: tempo -> Permite generar métricas a partir de trazas.

IMPLEMENTACION DE OPENTELEMETRY (NORMAL) AL PROYECTO

Dirigirse al repositorio oficial y descargar la versión mas reciente de este agente (deberá 
ser un archivo .jar): 
https://github.com/open-telemetry/opentelemetry-java-instrumentation?tab=readme-ov-file

El archivo se mueve a la raíz del proyecto y se ejecuta el siguiente comando.

OTLP = OpenTelemetry Protocol
Es el protocolo estándar de OpenTelemetry para enviar: traces, métricas y logs.

java -javaagent:opentelemetry-javaagent.jar \
-Dotel.service.name=mi-app \
-Dotel.traces.exporter=otlp \
-Dotel.metrics.exporter=none \
-Dotel.logs.exporter=none \
-Dotel.exporter.otlp.endpoint=http://localhost:4317 \
-Dotel.exporter.otlp.protocol=grpc \
-jar ./target/app-1.0.0.jar -> Levanta la aplicación pero antes carga un agente que la 
	instrumenta automáticamente.
	- -javaagent:nombreAgente.jar -> Antes de arrancar mi aplicación, carga este agente.
		Este agente intercepta llamadas HTTP, HDBC, redis, clientes HTTP.
	- -Dotel.service.name=mi-app -> Define el nombre lógico del servicio.
		En grafana aparecerá este nombre.
	- -Dotel.traces.exporter=otlp -> Para que la app exporte trazas con el mecanismo de 
		exportación llamado OTPL.
	- -Dotel.metrics.exporter=none -> Desactiva exportación de métricas.
		Porque Tempo solo recibe traces.
		Si no lo desactivas, el agente intentará enviarlas y puede generar errores.
	- -Dotel.logs.exporter=none -> Desactiva exportación de logs.
		Mismo motivo.
		Si quisieras logs necesitarías algo como: Loki o algo similar.
	- -Dotel.exporter.otlp.endpoint=http://localhost:4317 -> Para que envié todo lo que se
		haya configurado (trazas, métricas y/o logs) a determinada ruta.
		Aquí escucha tempo.
	- -Dotel.exporter.otlp.protocol=grpc -> Indica como se envían los datos que se hayan
		configurado (trazas, métricas y/o logs) con el exporter otlp.
	- -jar ./target/app-1.0.0.jar -> Indicar donde vive el jar de la aplicación.
		Evidentemtne se debió haber generado el .jar antes de usar este comando.

Para confirmar que todo este bien, los logs de la aplicación se deben de ver limpios.

IMPLEMENTACION DE OPENTELEMETRY (COLLECTOR) AL PROYECTO
 
Es un proxy inteligente de telemetría.
Collector puede enviar cada señal (trazas, logs y/o métricas) a un diferente destino: Traces → 
Tempo, Métricas → Prometheus y Logs → Loki.
Tiene un sampling inteligente pe: Enviar solo 20% de traces, Enviar 100% si son errores ó Enviar
100% si tardan más de 1s.

El Collector es opcional y actúa como intermediario entre las aplicaciones instrumentadas con 
OpenTelemetry y los backends de observabilidad (grafana). No reemplaza al agente OpenTelemetry; 
lo complementa agregando capacidades como procesamiento, enrutamiento, buffering y sampling 
avanzado.

Es importante usar collector en prod porque combate varios problemas:
1. si hay varias apps enviando sus datos directamente a tempo no se podrá limitar el trafico, ni 
filtrar spans y ni se podrá reducir el volumen, con collector si se puede lograr todos eso.
2. Si tempo se cae, las trazas se pierden pero si esta collector algunas trazas se puede guardar
un rato en collector.
3. mejora eficiencia porque collector envia bloques de spans.

3200), los demás puertos se quitan. Primero se hace esto porque el collector se conectara a 
tempo.

Se debe crear un archivo llamado otel-collector-config.yaml

Si el Collector fuera una fábrica:
- Receivers = puerta de entrada
- Processors = área de transformación
- Exporters = camiones que llevan mercancía

receivers: -> Configuraciones para recibir datos (traces, metrics o logs) de otro sistema.
  otlp_grpc:
    protocols:
      grpc: 
        endpoint: 0.0.0.0:4317 -> Crea una puerta de entrada para recibir datos otlp 
		con el protocolo grpc.
		Por convencion este protocolo usa el puerto es 4317 y se debe especificar
		que escuche en todas las interfaces de red.
		Debe escuchar en todas las interfaces de red para permitir conexión de 
		otros contenedores u otros hosts.
		Mas adelante cuando se levante la app con el agente, se indicara que se 
		envíen los datos aquí. ***

processors: -> Los processors modifican o procesan datos antes de enviarlos.
	Los processors se ejecutan en orden (de izq a der, según se hayan metido en el 
	arreglo que se le pasa a los processors).
	Configuraciones para esta opción:
	Configuraciones para loki y tempo ------------------------------------
	- batch: -> Agrupa muchas trazas/logs/métricas y después las envía juntos.
		Configuraciones para esta opción:
		En pruebas los valores recomendados son: 1, 64 y 128 respectivamente y para 
		producción son: 5, 512 y 1024 respectivamente.
		- timeout: ns -> Si no se llena el batch, el collector espera máximo n
			segundos antes de enviarlo.
		- send_batch_size: m -> Cuando se acumulan m spans, se envían 
			inmediatamente.
		- send_batch_max_size: z -> Proteje contra batches gigantes.
			Si llegan mas de m spans al mismo tiempo, lo máximo que se enviara
			serán z spans.
	- probabilistic_sampler:
    	     sampling_percentage: n -> Guarda solo el n% de las trazas aleatoriamente.
	- resource: -> Para agregar enriquecimiento a los datos. Significa agregar información 
		extra a las trazas, métricas o logs antes de enviarlos.
		Muchas veces la aplicación no incluye toda la información útil, entonces el 
		collector la agrega automáticamente.
    	     attributes:
      		- - key: nombreClave
       		  value: valor
        	  action: tipoAction -> Crea una clave y valor que será agregado a los datos.
			Tipos de accion:
			- insert -> Agrega el atributo solo si no existe.
			- upsert -> Inserta o reemplaza.
			- delete -> Elimina un atributo.
	- memory_limiter: -> Este processor protege al collector para que no se quede sin 
		memoria RAM y no se caiga, porque cuando tempo se cae el collector mantiene
		los datos en memoria, si no se configura esto el collector usara toda la
		memoria ram.
		Configuraciones para esta opción:
		- limit_percentage: 70 -> Si el proceso usa más del 70% de la memoria disponible
			empieza a rechazar datos
		- spike_limit_percentage: 25 -> si la memoria aumenta demasiado rápido
			limita cuánto puede crecer en ese pico.
  		- check_interval: 5s -> Cada n segundos el collector revisa cuanta memoria 
			esta usando
	Configuraciones para loki ------------------------------------
	- attributes:
    	    actions:
            - key: loki.attribute.labels
              action: upsert
              value: nombreCampo1, nombreCampo2, nombreCampoN -> Tomando campos existentes, los
		convierte en campos labels.

exporters: -> Configuraciones para enviar los datos a otro sistema.

IMPLEMENTACION DE LOKI

Loki: Es un sistema de agregación de logs creado por Grafana Labs.
Su objetivo es: Guardar, organizar y consultar logs Pero de una forma más ligera y barata que otros sistemas.

Loki también puede crear sus propias reglas para mandar alertas, pero hoy se recomienda que las reglas y 
alertas las maneje grafana. Entonces se puede ommitir el componente ruler y la creación de la carpeta
rules_directory. **

Un stream es un conjunto de logs con exactamente la misma combinación de labels.
Ejemplo real:
{app="api",env="prod",pod="abc"}

El mayor problema de Loki es que un mismo label tenga una cardinalidad enorme porque esto provoca muchos
streams. ***

Configurar la app ----------------------------

Hay dos tipos con loki:
- labels: Se usan para indexar.
	No debe tener cardinalidad alta porque esto genera millones de combinaciones. ****
	Los valores comunes seria: el nombre del ambiente, el nombre de la app, el nivel de 
	información (error, info, debug, etc)
- campos: Son datos que van dentro del json.
	Evidentemente no se agrega datos como contraseñas, tokens, tarjetas, etc.
	En el collector decides qué campos del logJson se vuelven labels. ****

Se crea el package llamado config y dentro de este se crea la clase 'RequestLoggingFilter'.
Esta clase es un filtro de Spring que intercepta todas las peticiones HTTP y con base a esas peticiones
se crean campos personalizados que se agregaran a los logs de loki, algunos de los campos incluyen:
nombre del ambiente, endpoint, método http, status, cuerpo de la request, entre otros.

Esa clase se divide en tres partes dentro del método doFilterInternal:
1. Antes de filterChain.doFilter(...): Son las acciones que se ejecutan antes de que la request
	llegue al siguiente filtro o al controller.
2. Llamada a filterChain.doFilter(...): Es lo que permite que la request continúe su flujo.
	Puede ir a: otro filtro (si hay más en la cadena) o al controller (si ya es el último 
	filtro).
3. Después de filterChain.doFilter(...): lógica posterior cuando ya terminó toda la cadena (otros
	filtros y el controllador).
Nota: Hay algunos valores que están disponibles hasta determiado punto del flujo.

@Component
class RequestLoggingFilter : OncePerRequestFilter() {} -> Se crea la clase y se convierte en un componente 
	de spring boot.
	El hacer que se herede de la clase OncePerRequestFilter implica que el filtro se ejecuta una sola 
	vez por cada request HTTP.

@Throws(ServletException::class, IOException::class)
override fun doFilterInternal(
    request: HttpServletRequest,
    response: HttpServletResponse,
    filterChain: FilterChain
) {...} -> Método principal del filtro.
	Aquí se define qué hacer antes, durante y después de una request HTTP.
	Spring lo ejecuta automáticamente por cada request.
	- request: HttpServletRequest -> Representa la petición HTTP que llega.
		Aquí puedes obtener: método http, headers, body, parámetros, URL, etc.
	- response: HttpServletResponse -> Representa la respuesta que vas a enviar.
	- filterChain: FilterChain -> Controla por donde pasara la request.
	- @Throws(...) -> Para indicar que este método puede lanzar estas exceptiones.
		Es necesario agregar esto Porque el método original que estás sobrescribiendo (de 
		OncePerRequestFilter) ya declara esas excepciones.

val wrappedRequest = ContentCachingRequestWrapper(request) -> Envuelve la request original en un wrapper que:
	permite leer el body (cuerpo) de la request múltiples veces.
	Si no se usa esto, entonces el controller ya no podrá leer el body y por tanto el flujo fallara.

wrappedRequest.method -> Devuelve el nombre del método http de la request.

wrappedRequest.requestURI -> Devuelve el valor de la url sin los query params, los path variables si los incluye.***

wrappedRequest.queryString -> Devuelve el valor de todos los query params junto con sus respectivos valores.

wrappedRequest.getHeader("nombreHeader") -> Extrae el valor asociado a determinado header.

wrappedRequest.headerNames -> Devuelve le nombre de todos los headers.

wrappedRequest.getParameter("nombreQueryParam") -> Devuelve el valor asociado a determinado query param.

wrappedRequest.contentLength -> Devuelve el tamaño del body.

try {
    filterChain.doFilter(wrappedRequest, response)
} catch (ex: Exception) {
    throw ex
} finally {...} -> Para que la request y la response vayan al siguiente filtro o 
	en su defecto a controller que le corresponde ir.
	Si no se llama a esta función la request se quedara ahí y nunca llevara al controller. ***
	Esta instrucción va en un try catch porque ahí ocurren otros filtros, el controller, etc y
	eso puede provocar excepciontes. En el catch es necesario que se lance la exception que 
	se capture porque si no spring no se enterara de la exception. 

wrappedRequest.contentAsByteArray -> Devuelve el body de la request.
	El body solo estará disponible después de doFilter. ****

response.status -> Devuelve el status de la respuesta.

val endpoint = wrappedRequest.getAttribute(
    HandlerMapping.BEST_MATCHING_PATTERN_ATTRIBUTE
) as? String ?: wrappedRequest.requestURI -> Intenta obtener el endpoint mapeado por Spring (con 
	patrones) Si no puede, usa la URL real (requestURI).
	Es super importante usar este valor en loki porque al ser un patron tiene baja 
	cardinalidad perfecto para los labels. *****
	Este valor solo existe después de que Spring resuelve el controller. ***
	- HandlerMapping.BEST_MATCHING_PATTERN_ATTRIBUTE -> Representa El patrón del endpoint que 
		mejor coincidió con la request.

MDC.put("nombreCampo", valor); -> Para crear y asignar un valor al logJson.
	MDC significa Mapped Diagnostic Context.
	Permite agregar información contextual al hilo actual.
	El logger agrega automáticamente como campo el nivel de log (cuando se hace log.nivel()). 
	Mientras que OpenTelemetry agrega automáticamente el trace_id para que se puede 
	correlacionar un trace_id de tempo con un log de loki y viceversa.

MDC.clear() -> Elimina TODOS los valores del MDC del hilo actual.
	Es indispensable usar esta instrucción dentro de un finally para que se ejecute siempre
	y evite contaminar datos.

ARQUITECTURAS PRODUCTIVAS

Esta teoría es de acuerdo a chatgpt.

Modelo: collector distribuido + grafana central ------------------

Cada EC2 Proyecto
 └─ OpenTelemetry Collector (local)
        │
        ▼
EC2 Observabilidad
 ├─ Tempo
 ├─ Prometheus
 ├─ Loki
 └─ Grafana

Aquí:
Cada proyecto tiene su Collector local
Ese Collector envía datos al backend central

Este modelo se usa mucho en empresas medianas.

Modelo: Todo separado ---------------------------------------

EC2 1 → Grafana
EC2 2 → Tempo
EC2 3 → Prometheus / Mimir
EC2 4 → Loki
EC2 5 → Collector cluster
Ec2 6 - N -> Cada app en una instancia diferente.

Este modelo es el ideal pero es el mas caro.


Modelo: Personalizado ---------------------------------------

Con base en los modelos que ha propuesto la ia se pensó un modelo mejor.

EC2 1 → Grafana
EC2 2 → Tempo
EC2 3 → Prometheus / Mimir
EC2 4 → Loki
EC2 5 → Collector cluster
Ec2 6 - N -> Cada app con su propio collector en una instancia diferente.

Este modelo es igual de caro que el ideal pero con la diferencia es que cada app tiene su 
propio collector con la finalidad de filtrar información y que al collector maestro no le 
lleguen tantos datos.


CONSTRUCCION DEL DOCKER COMPOSE

No es necesario usar depends_on hacia Grafana porque Grafana solo es la capa de visualización. Es 
decir, servicios como Prometheus, Loki o Tempo deben funcionar de manera totalmente independiente 
recolectando y almacenando métricas, logs y trazas, aunque Grafana no esté disponible.

Asi mismo, lo ideal es que todas las herramientas puedan estar levantadas de manera independiente
para que puedan seguir recolentando datos. En resumen no se debe usar depends on en ninguna 
herramienta.


CONFIGURACION DE LAS INSTANCIAS

Hay dos instancias: la instancia con la app y la instancia de observabilidad (con las herramientas: 
grafana, loki, promethues, collector y tempo).

En la instancia de observabilidad se abren las siguiente reglas de entrada:
	- puerto 4317 para que el collector reciba datos.
	- puerto 3000 para poder acceder a grafana.

Idealmente la configuración en la gui de aws debe ser la siguiente: 
Campo	Collector	Grafana
Puerto	4317		3000
Tipo	TCP		TCP
Source	instancia APP	tu IP

En la instancia de la app, Probablemente NO necesitas abrir ninguna regla de salida (outbound)
porque Por defecto, los Security Groups tienen esto: Outbound: 0.0.0.0/0 → ALLOW y Eso significa:
la instancia puede salir a cualquier IP en cualquier puerto. 

Si fuese el caso de tener que abrir una regla de salida, la configuración en aws debe ser la 
siguiente:
Campo		Valor
Type		Custom TCP
Port		4317
Destination	IP de la instancia de observabilidad

*En la instancia de la app*
nc -zv IP_INSTANCIA_OBSERVABILIDAD 4317 -> Verifica que la regla de salida en la instancia con la 
	app funcione y que la regla de entrada en la instancia de observabilidad funcione.
	Si este comando falla no se puede determinar que fallo exactamente.

Para debuggear *Dentro de la instancia de observabilidad*: docker ps y docker logs colllector

ss -tulnp | grep 4317 -> Verifica si hay un proceso escuchando en el puerto 4317 dentro de la instancia.
