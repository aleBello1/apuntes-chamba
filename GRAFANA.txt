METRICAS 

http_server_requests_seconds_count -> Es un counter. Contiene el numero total acumulado de 
	requests HTTP que tu aplicación ha recibido.
	Empieza en 0 cuando arranca la app y nunca disminuye.
	Cuando la app se reinicia, el counter vuelve a 0, Prometheus detecta ese reset 
	automáticamente
	Parámetros/labels que trae:
	- method -> Método HTTP (GET, POST, etc.)
	- status -> Codigo de status http de la request (200, 404, 500, etc)
	- uri -> Url del servicio
	- outcome -> Nombre del codigo http (SUCCESS, CLIENT_ERROR, SERVER_ERROR)
	- exception -> Nombre de la excepción si ocurrió error (puede estar vacío)
	

CONSULTAS PROMQL

Promql significa lenguaje de consultas de prometheus.

nombreMetricaZ{nombreParametroX operador valorX} -> Para condicionar que la metrica deba tener 
	cierto valor en determinado parámetro.
	-  nombreParametro1 operador valor1, nombreParametro2 operador valor2, nombreParametroN 
		operador valorN -> Se colocan comas para condicionar por varios labels, la coma
			seria equivalente a un and.
	Ejemplo: {status=~"4.."} seria para seleccionar todos los datos que en el parámetro 
		status inicie con 4xx.
	Operadores: 
	- != → Diferente
	- =~ -> Para indicar que lo siguiente será una expresión regular.

nombreMetricaZ[tiempo] -> Para aplicar un range vector.
	Significa Toma los valores de esta métrica en los últimos n unidad de tiempo.
	Ejemplo: 5m seria los últimos 5 minutos.
	Unidades:
	- s	segundos
	- m	minutos
	- h	horas
	- d	días
	- w	semanas
	- y	años

Funciones -----------------------------

rate(...) -> Calcula la velocidad promedio por segundo a la que está creciendo un contador.
	Devuelve un valor por segundo.
	Se usa principalmente para métricas tipo Counter.

by() -> Permite hacer un group by.

increase(metrica[tiempo]) -> Calcula cuánto aumentó el contador dentro del rango de tiempo que 
	le pasas. **
	Solo tiene sentido en metricas de tipo Counters.
	Ejemplo si increase(contador[5m]) y si el primer valor es a las 10.00am y hubo 30 	
	mientras que a las 10.05am hubo 47 entonces se graficara a las 10.00am un 30 y a las 
	10.05am se graficara un 17.

Instrumentar una aplicación significa: Agregarle la capacidad de medir y reportar lo que está haciendo 
internamente.
Si un servicio está instrumentado:
- Mide cuánto tardó.
- Mide cuántas veces se llamó.
- Mide si fue error 500.
- Guarda un trace con todos los pasos internos.
- Puede reportar métricas, logs y trazas.

OpenTelemetry es un estándar open source para instrumentar aplicaciones.
Puede enviar logs, métricas y traces.

Grafana tempo: Es la base de datos donde se guardan los requests de tu aplicación (las trazas).
Una traza es El recorrido completo de una petición dentro del sistema.
Span: Una operación individual dentro del trace

El flujo ideal de herramientas debe ser el siguiente:

App (Combinado con el agente OpenTelemetry)
        ↓
Collector (receiver → processor → exporter)
        ↓
Tempo (almacena trazas)
        ↓
Grafana (visualiza)

	- -Dotel.service.name=mi-app -> Define el nombre lógico del servicio.

IMPLEMENTACION DE OPENTELEMETRY (COLLECTOR) AL PROYECTO
 
Es un proxy inteligente de telemetría.
Collector puede enviar cada señal (trazas, logs y/o métricas) a un diferente destino: Traces → 
Tempo, Métricas → Prometheus y Logs → Loki.
Tiene un sampling inteligente pe: Enviar solo 20% de traces, Enviar 100% si son errores ó Enviar
100% si tardan más de 1s.

El Collector es opcional y actúa como intermediario entre las aplicaciones instrumentadas con 
OpenTelemetry y los backends de observabilidad (grafana). No reemplaza al agente OpenTelemetry; 
lo complementa agregando capacidades como procesamiento, enrutamiento, buffering y sampling 
avanzado.

Se vuelve a crear el contenedor de tempo pero ahora solo exponiendo el puerto principal (el puerto
3200), los demás puertos se quitan. Primero se hace esto porque el collector se conectara a 
tempo.

	todo esta ok.

Se debe crear un archivo llamado otel-collector-config.yaml

Si el Collector fuera una fábrica:
- Receivers = puerta de entrada
- Processors = área de transformación
- Exporters = camiones que llevan mercancía

receivers: -> Configuraciones para recibir datos de otro sistema.
  otlp_grpc:
    protocols:
      grpc: 
        endpoint: 0.0.0.0:4317 -> Crea una puerta de entrada para recibir datos otlp 
		con el protocolo grpc.
		Por defecto este protocolo usa el puerto es 4317 y se debe especificar
		que escuche en todas las interfaces de red.
		Mas adelante cuando se levante la app con el agente, se indicara que se 
		envíen los datos aquí. ***

processors: -> Los processors modifican o procesan datos antes de enviarlos.
	Configuraciones para esta opción:
	- batch: -> No envía cada span inmediatamente.
		Los agrupa en pequeños lotes.
		Los envía juntos.
	- probabilistic_sampler:
    	     sampling_percentage: n -> Guarda solo el n% de las trazas aleatoriamente.

exporters: -> Configuraciones para enviar los datos a otro sistema.
	Configuraciones para esta opción: 
	- otlp_grpc: -> En teoría este nombre es indiferente porque es como si fuera el 
		nombre de un objeto donde lo importante es su conteneido sin embargo
		la documentación recomienda este nombre.
		- endpoint: nombreContenedorTempo:4317 -> Para que se envíen datos al 
			servicio llamado tempo el cual esta en el puerto 4317.
    	    	- tls:
		     insecure: true -> No verifiques certificados TLS.
			Permite conexión sin validación segura.

service:
  pipelines: -> Define los pipelines, el cual se encargan de recibir, procesar y exportar
	datos.
	Configuraciones para esta opción:
	- traces: -> Indica el pipeline de las trazas.
		Configuraciones para esta opción:
		- receivers: [nombreReceiver1, nombreReceiver2, nombreReceiverN] -> Para
			indicar que receiver's de los que se configuraron en la etiqueta
			receiver llegaran las trazas al collector.
			No importa que dentro de la etiqueta exporters se tenga uno con 
			el mismo nombre porque no se definen los nombres de manera
			global, cada sección es indepentiente, entonces solo buscara 
			los receivers definidos dentro de la etiqueta recervers.
			Lo común seria colocarle solo otlp para indicar que recibirá 
			traces de tipo OTLP. 
      		- processors: [nombreProccesor1, nombreProccesor2, nombreProccesorN] -> Se le pasa 
			todos los processors que queremos que tenga esta señal.
			Estos proccesor se debieron definir anteriormente dentro de la etiqueta
			proccessors. ***
      		- exporters: [nombreExporter1, nombreExporter2, nombreExporterN] -> Para
			indicar que exporter's de los que se configuraron en la etiqueta
			exporters se enviaran las trazas del collector hacia afuera.
			Se envían al exporter OTLP (Tempo)

docker run -d --name collector -p 4317:4317 --network red otel/opentelemetry-collector-contrib:latest

	- -p 4317:4317 -> Este es el puerto principal del collector y es para recibir con el

IMPLMENTACION DE LOKI + PROMTAIL

Loki: Es un sistema de agregación de logs creado por Grafana Labs.
Su objetivo es: Guardar, organizar y consultar logs Pero de una forma más ligera y barata que otros sistemas.

Promtail: es el agente recolector de logs oficial de Loki.
Su función es: Leer logs, Agregar etiquetas (labels) y Enviarlos a Loki.
No guarda los logs, solo los transporta.

Flujo de los dos: App escribe en los logs → el Archivo cambia → Promtail detecta cambios → 
Promtail agrega labels → se los Envía a Loki → Grafana consulta a Loki

Modelo: collector distribuido + grafana central ------------------

Cada EC2 Proyecto
 └─ OpenTelemetry Collector (local)
        │
        ▼
EC2 Observabilidad
 ├─ Tempo
 ├─ Prometheus
 ├─ Loki
 └─ Grafana

Aquí:
Cada proyecto tiene su Collector local
Ese Collector envía datos al backend central

Este modelo se usa mucho en empresas medianas.

Modelo: Todo separado ---------------------------------------

EC2 1 → Grafana
EC2 2 → Tempo
EC2 3 → Prometheus / Mimir
EC2 4 → Loki
EC2 5 → Collector cluster
Ec2 6 - N -> Cada app en una instancia diferente.

Este modelo es el ideal pero es el mas caro.
