


/////////////////////////////////ADMINISTRACION DE DASHBOARDS////////////////////////////////


DATASOURCE

Es como dar de alta una db de la cual grafana pueda consultar y con base a esos datos crear
dashboards y alertas.

Conectar grafana con algún data source: Apartado izq. connections -> data sources -> add new 
	data source -> seleccionar determinado data source -> Ingresar en la url el valor 
	http://nombreContenedor:puertoContenedor, se usa el nombre del contenedor
	que almacena a la herramienta de interés y su puerto -> save and test.

Eliminar determinado data source: Apartado izq. connections -> data sources -> Seleccionar 
	determinado data source -> irse hasta la parte inferior -> delete.


DASHBOARDS 

Grafana tiene dashboards ya hechos por la comunidad. En lugar de crear gráficas desde cero, 
puedes importar esos dashboards.

Uso del apartado izq. llamado explore: Sirve para Probar queries, Debuggear métricas, Ver
	resultados rápidos y Experimentar. 
	Aquí NO se guardan dashboards. ***

Uso del apartado izq. llamado dashboards: Es donde sí puedes Crear paneles, Guardarlos, 	
Compartirlos y Versionarlos. 

Activar el autorefrescado en las queries: Apartado izq. explore -> Alado del button de run query
	presionar la flecha hacia abajo y seleccionar cada cuantos segundos queremos el 
	autorefrescado.

Crear dashboard: Apartado izq. dashboards -> new -> new dashboard -> add visualization
	-> hacer las queries al gusto -> en la parte sup derecha sabe dashboard -> asginar 
	nombre y descripción al dashboard.

Agregar una nueva grafica en un dashboard existente: Apartado izq. dashboards -> seleccionar 
	determinado dashboard -> parte superior buscar el button edit -> add -> visualization

Visualizar los dashboards existentes: Apartado izq. dashboards -> aquí se enlistaran todos los
	dashboards existentes.

Editar un dashboard existente: Apartado izq. dashboards -> aquí se enlistaran todos los
	dashboards existentes -> click sobre determinado dashboar -> edit.

Crear dashboard dinámico: En vez de crear 10 dashboards iguales (uno por endpoint o servicio) 
	se crean dashboard con variables las cuales se encargan de cambiar ciertos datos 
	específicos por cada dashboard.
	*Estando en la edición de un dashboard* -> En la part sup derecha -> back to dashboard
	-> parte sup settings -> apartado variables -> add variable -> asignarle nombre y label
	(el label es como aparecerá en la gui) a la variable -> query type: label values -> 
	metric: seleccionar determinada metrica -> label: seleccionar determinado label de la 
	metrica (el cual significa dame todos los posibles valores que tenga determinado label 
	de cierta metrica) -> ya después nos podemos regresar a determinado dashboard e irle 	
	implmentando a las queries la nueva variable. Para usar una variable es con la sintaxis 
	$nombreVariable.


Creación de graficos ----------------------------------------------------

Visualizar el estado de salud de la app (ver si esta arriba o abajo): dashboard -> prometheus -> 
	metrica: up -> job: ingresar el nombre de la herramienta.

Graficar la ram de un proceso: Apartado izq dashboards -> new -> new dashboard -> add visualization
	-> prometheus -> meter la consulta que represente la división de la ram usada entre el 
	limite de ram permitida multiplicada por 100 -> tipo de grafica: time series.
	- Colocar el porcentaje: Apartado derecho buscar el apartado que dice standard options
		-> misc -> percent.
	- Agregar valores de calor: Apartado derecho buscar el apartado que dice thresholds ->
		add threshold.
	- Consulta: (sum by (job) (jvm_memory_used_bytes{area="heap", job=~"job1|job2|jobN"}) 
		/ sum by (job) (jvm_memory_max_bytes{area="heap", job=~"job1|job2|jobN"})) * 100

Graficar la cpu de un proceso: Apartado izq dashboards -> new -> new dashboard -> add visualization
	-> prometheus -> ingresar la metrica de cpu -> tipo de grafica: time series.
	- consulta: avg by (job) (process_cpu_usage{job=~"job1|job2|jobN"}) * 100

Para visualizar en el tiempo el estado del circuito: Apartado izq. dashboards -> new -> new 
	dashboard -> seleccionar como data source: prometheus -> en metric buscar: 
	'resilience4j_circuitbreaker_state' -> run query.
	Esto nos mostrara en el tiempo todos los estados del circuito: disable, half-open,
	open and close.
	Podremos dar click sobre determinada leyenda para observar solamente a ese vector.
	- Para ver solo el estado activo del circuito: click en operations -> equal to -> value:
		1 -> run query.
	- Editar como aparecen las leyendas: click sobre el apartado desplegable llamado options
		-> seleccionar custom ->  En el legend se debe usar interpolación para 
		seleccionar el valor de determinado atirbuto de la leyenda. Pe si la leyenda 
		original es: nombreMetrica(parametro1=valor1, parametro2=valor2, 
		parametroN=valorN) si se desea seleccionar el valor asociado al parámetro 2, se 
		debe escribir en legend {{parametro2}}.
		Ademas se puede concatenar valores con la concatenación por ejemplo: El estado
		del circuito es: {{state}}.



MEDICION DE ERRORES EN APLICACIONES

Existen 3 formas principales de medir errores: 
- Contar logs de tipo ERROR (para ello se Loki + promtail). Evidentmente esto depende 100% de 
	que realmente estés logueando errores.
- Medir códigos HTTP. 
- Desconocido aun.


Medir código http --------------------------------------------

En si, en este punto ya todo estará configurado pero se pueden seguir unos pasos para comprobar
que se esta consumiendo la metrica de interés.

Ingresar a http://localhost:5000/actuator/prometheus y buscar que exista: 
http_server_requests_seconds_count.

Ahora ir a prometheus (http://localhost:9090) y buscar lo mismo.

Lo común es mostrar los códigos de status http 4xx y 5xx en un mismo dashboard y se crear un 
dashboard por cada servicio que tenga la app.

Crear un dashboard que muestre el porcentaje de errores 5xx enpoint: Apartado izq. dashboards
	-> new -> new dashboard -> add visualization -> En datasource prometheus 
	- Consulta: (sum by (uri) (rate(http_server_requests_seconds_count{status=~"5..", 
		job="ws-mapfre-dev"}[3m])) / sum by (uri) 
		(rate(http_server_requests_seconds_count{job="ws-mapfre-dev"}[3m]))) * 100

Crear un dashboard que muestre el porcentaje de errores 4xx enpoint: Apartado izq. dashboards
	-> new -> new dashboard -> add visualization -> En datasource prometheus 
	- Consulta: (sum by (uri) (rate(http_server_requests_seconds_count{status=~"4..", 
		job="ws-mapfre-dev"}[3m])) / sum by (uri) 
		(rate(http_server_requests_seconds_count{job="ws-mapfre-dev"}[3m]))) * 100

ALERTAS

Crear un punto de contacto: Es definir por cual medio grafana nos enviara alertas: email, sms,
	slack, etc.
	Apartado izq. alerting -> contact points:
	- Crear el contacto point para email: Primero se debe crear un servidor SMTP (Simple Mail
		Transfer Protocol), esto es un servicio que se encarga de enviar correos 
		electrónicos.
		Grafana no “envía correos”. Grafana le dice a un servidor SMTP: oye manda este
		correo a tal persona. 
		El SMTP es de paga y en teoría la empresa debe proporcionar las credenciales pero
		para pruebas locales se puede usar nuestro propio correo.
		créate contact point -> asginar name -> integration: email -> asginar el correo o 
		los correos -> si se ingresaron muchos correos palomear la opción single email
		para enviar un solo correo y no multiples -> test (solo si necesita hacer una
		prueba) -> sabe contact point.
	- Crear el contacto point para slack: Ingresar a este link: https://api.slack.com/apps -> 
		créate new app -> Ingresar el nombre que tendrá la app -> sig -> En el menú
		izquierdo: Incoming Webhooks -> activar Activate Incoming Webhooks -> Add New 
		Webhook to Workspace -> Hasta aquí se dejo este paso porque se necesitaba que 
		algún administrador de nuestro canal de slack aprobara a este bot/app.

Generar contraseña de aplicación en Gmail: Esto es para que desde nuestro correo se envíen las
	alertas, nadamas es para pruebas, pero para producción lo ideal es usar un SMTP. **
	Importante: necesitas tener activada la verificación en 2 pasos.
	- Activar verificación en 2 pasos: Ir a Gmail -> click sobre nuestra foto -> click sobre
		nuestrea cuenta -> seguridad y acceso -> verificación de 2 pasos.
	Ir a este enlace:  https://myaccount.google.com/apppasswords -> asginar nombre a la 
	app -> crear -> nos arrjara la contraseña para esa app (se debe usar sin espacios en
	blanco aunque Gmail nos la de con espacios en blanco***). 

Crear alerta: Apartado izq. alerting -> alert rules -> new alert rule -> elegir el data source
	-> la metrica y la condición que disparara la alerta -> run query para asegurar que sea
	una query valida -> elegir en que carpeta estara almacenada la alerta -> seleccionar el 
	grupo de alertas al que pertenecera la alerta -> elegir el pending period (el cual 
	significa que si la condición es true dudante cuanto tiempo se esperara para lanzar la 
	alerta) -> definir el keep firing for (el cual sginifica tiempo extra en que la alerta 
	se queda encendida aunque el problema ya se resolvió) -> seleccionar el contact point -> 
	agregar el subject y cuerpo del correo -> sabe.
	Se debe considerar que en un correo el subjet dira algo como: {nombre alerta} - {nombre 
	folder} ******

silece: Es Ignorar temporalmente alertas que cumplan ciertas etiquetas.
	No apaga la alerta, No cambia la evaluación y Solo evita notificaciones.
	Cada alerta tiene labels, por ejemplo: alertname="AppDown"; job="mi-app";
	instance="10.0.0.1:8080" y el silence dice si una alerta tiene estos labels entonces 
	no notificar.


VALORES RECOMENDADOS PARA PENDING PERIOD

for: 1m (rápido). Usado cuando el problema no puede esperar.
	Se usa para detectar servicios caídos (up == 0).
for: 2m (balanceado). Usado para evitar falsos positivos.
	Por ejemplo para cpu, ram y latencia.
	Se usa para alertas criticas pe cuando se supera un 90% de ram o cpu. ****
for: 5m (balanceado). Usado para evitar falsos positivos.
	Por ejemplo para cpu, ram y latencia.
	Se usa para alertas warning pe cuando se supera un 80% de ram o cpu. ****
for: 10m+ (lento): Se usa para monitorear el espacio en disco.

Apps -----------------------------------------------------------

Avisar cuando el corto circuito este abierto por mas de 1 min.

Avisar cuando la ram supera el 80% por mas de 5 min y agregar avg_over_time(up{job="nombreApp"}[2m]) > 0.9

Avisar cuando la ram supera el 90% por mas de 2 min y agregar avg_over_time(up{job="nombreApp"}[2m]) > 0.9

Avisar cuando el cpu supera el 80% por mas de 5 min y agregar avg_over_time(up{job="nombreApp"}[2m]) > 0.9

Avisar cuando la cpu supera el 90% por mas de 2 min y agregar avg_over_time(up{job="nombreApp"}[2m]) > 0.9

La condición avg_over_time(up{job="nombreApp"}[2m]) > 0.9 sirve para condicionar a que La app 
estuvo UP el 90% del tiempo durante los últimos 2 minutos. Esto se hace porque si cuando la app se 
reinicia la RAM y la CPU vuelve subir de golpe y esto provoca falsos positivos, entonces 
condiciona a que si la app esta viva y sube la RAM o CPU no es un falso negativo.

Avisar cuando el porcentaje de errores 5xx por cada endpoint en ventanas de 5m haya superado cierto valor y
para que no haya falsos positivos se agrega la condición de que hay un numero mínimo de request (por ejemplo
lanzar la alerta solo si hubo un 70% de errores y al menos unas n request).







( sum by (uri) ( rate(http_server_requests_seconds_count{ job="ws-mapfre-prod", status=~"5..", uri!~"/health|/metrics|/actuator.*|/webjars.*|/swagger-ui.*|/v3.*" }[5m]) ) / sum by (uri) ( rate(http_server_requests_seconds_count{ job="ws-mapfre-prod", uri!~"/health|/metrics|/actuator.*" }[5m]) ) ) > m -> Un endpoint tiene más de m% de errores 5xx en los últimos 5 minutos


sum by (uri) ( increase(http_server_requests_seconds_count{ job="ws-mapfre-prod", uri!~"/health|/metrics|/actuator.*|/webjars.*|/swagger-ui.*|/v3.*" }[5m]) ) -> Z # Solo si hay al menos z request en los últimos 5 min.
