CONFIGURACION E IMPLEMENTACION DE APP + TEMPO

Si solo se ocupa openTelemetry + tempo (sin el collector en medio de los dos), primero se debe crear
el contenedor de tempo porque openTelemetry se conectara a él.

En la raíz del proyecto se debe crear un archivo llamado tempo.yaml y tener ciertas configuraciones.

Las siguientes configuración solo sirven para tempo versión 2.7.1 (Para determinarlo nos metimos
al contenedor y ejecutamos tempo --version).

stream_over_http_enabled: true -> Permite que Tempo envíe resultados de consultas a Grafana usando 
	streaming HTTP.
	Al hacerlo con stream Tempo empieza a mandar datos mientras grafana todavía los está 
	leyendo. Esto provoca consultas más rápidas.

server:
	Configuraciones para esta opción:
	- http_listen_port: 3200 -> Abre un servidor HTTP en el puerto 3200.
		Ese puerto sirve para: Que Grafana se conecte a Tempo, Consultar trazas y Hacer 
		búsquedas.
		Este puerto NO es para que tu aplicación envíe trazas.
	- log_level: nivelLog
		- debug -> muy detallado
		- info	normal
		- warn	advertencias
		- error	errores

distributor: -> se encarga de recibir spans, validarlos y distribuirlos a los ingesters.
	- receivers: -> Define qué protocolos acepta Tempo para recibir trazas.
		- otlp: -> Para decir que se recibirá trazas usando el protocolo OTLP.
			Configuraciones para esta opción:
      			- protocols:
         			- grpc:
				    endpoint: 0.0.0.0:4317
         		 	  http: 
				    endpoint: 0.0.0.0:4318 -> Habilitar dos formas de recibir 
					trazas: via grpc (puerto 4317) y http (puerto 4318) en 
					todas las interfaces. Escuchar en todas las interfaces de red
					permite conexiones desde otros contenedores y otras máquinas.

ingester: -> Para configurar al intester.
	Los ingesters reciben las trazas desde el distributor y las preparan para almacenamiento.
	Configuraciones para esta opción:
	- trace_idle_period: ns -> Tempo espera n segundos sin recibir spans nuevos para 
		considerar que la traza está completa.
		Ejemplo: traceID 123 -> span1 -> span2 -> span3. Si en n segundos no llegan más 
		spans entonces Tempo cierra la traza.
	- max_block_bytes: 100 -> Tamaño máximo de bloque en memoria antes de escribirlo.
		Tempo no guarda cada span inmediatamente en disco.
		Primero los acumula en memoria y Cuando el bloque alcanza cierto tamaño o tiempo
		se guarda en el storage.
		En prod, el valor recomendado es 500MB (524288000 bytes) hasta 1GB y para pruebas
		se puede dejar entre 1MB a 10MB.
	- max_block_duration: 30m -> controla el tiempo máximo que un bloque puede permanecer 
		abierto.
		Este parámetro evita que un bloque permanezca abierto demasiado tiempo si el tráfico 
		es bajo.
		Entonces si el bloque no alcanza el max_block_bytes pero ya paso 
		max_block_duration el bloque se cierra.
		En prod el valor recomendado entre 30m y 1h.

storage: -> Define dónde se guardan las trazas.
  trace:
    backend: tipoUbicacion -> Indica el tipo de almacenado de las trazas.
	Ubicaciones:
	- local
	- s3
	- gcs
	- azure
	Configuraciones para esta opción:
	- wal:
	    path: rutaX -> Configura el wal (Write Ahead Log) y su ruta. Es un log de seguridad. Antes 
		de guardar datos en almacenamiento final tempo escribe en wal.
		Si tempo crashea se recupera desde wal.
	Si se uso local: -----------------------------
	- local: # se repite esta linea
	    path: /tmp/tempo/traces -> Indica la ruta en la maquina local donde se guardaran las
		archivos.
		Si tempo esta dentro de un contenedor y se quiere implementar un volumen de persistencia
		esta ruta debe coincidir con la ruta del volumen.
	
compactor: -> Optimiza los bloques almacenados.
  compaction:
	- block_retention: n[unidad_tiempo] -> Guarda trazas solo por n cantidad de tiempo después
		trazas que superen esta antigüedad será eliminado automáticamente.
		Si esto no se configura siempre se guardarán las trazas hasta que el disco se 
		llene.
		Unidades:
		- h	horas
		- m	minutos
		- s	segundos
	- compacted_block_retention: 1h -> Mantiene los bloques viejos un tiempo antes de 
		borrarlos.
		El valor en prod recomendado es entre 1h y 2h.

overrides:
  defaults: -> sirve para limitar cuántos datos podemos recibir de cada 
	cliente
	Configuraciones para esta opcon:
	- max_traces_per_user: x -> Define el número máximo de trazas activas que un usuario 
		puede tener en memoria.
		Si llega a x trazas activas, Tempo empieza a rechazar nuevas trazas.
    	- ingestion_rate_limit_bytes: y -> limita cuántos bytes por segundo puede enviar un 
		cliente.
		si lo supera Tempo empieza a responder: 429 Too Many Requests.

query_frontend:
  search:
    duration_slo: ns -> componente que optimiza consultas desde Grafana.
	divide consultas grandes, paraleliza búsquedas y cachea resultados.
	Para indicar que Las búsquedas deberían resolverse en máximo n segundos.

metrics_generator:
  registry:
    external_labels:
      source: tempo -> Permite generar métricas a partir de trazas.

Después crear el contenedor.

MSYS_NO_PATHCONV=1 docker run -d \
  --name tempo \
  -p 3200:3200 \
  -p 4317:4317 \
  -p 4318:4318 \
  --network red \
  -v volTempo:/traces \
  -v volTempoWal:/etc/tempo/tempo_wal \
  -v /c/users/alejandro/desktop/MAPFRE/ws-mapfre/tempo.yaml:/etc/tempo/tempo.yaml \
  ubuntu/tempo:2.7.1-24.04_stable -> Levanta un contendor con una imagen de tempo.
	Se debe de poner la ruta completa del archivo si se usa git bash.
	- -p 3200:3200 -> Puerto donde corre tempo.
	- -p 4317:4317 -> Puerto donde recibe trazas via grpc.
	- -p 4318:4318 -> Puerto donde recibe trazas via http.
	- /etc/tempo/tempo.yaml -> Archivo de configuración que carga el contendor
		al iniciar.
	- -v rutaCompleta/tempo.yaml:/etc/tempo/tempo.yaml -> Para meterle al contendor nuestra 
		configuración de tempo.yaml

Se debe usar específicamente esta imagen y esta versión, estas si mostraron buenos 
resultados.********

Para comprobar que se usa nuestro archivo
docker exec -it tempoDefinitivo sh
cat etc/tempo/tempo.yaml

Endpoints de tempo:

GET /metrics -> Exponen métricas de tempo.

GET /ready -> Para verificar si tempo esta vivo.

Agregar a tempo como data source en grafana: *En grafana* -> apartado izq. connections -> data 
	sources -> add new data source -> tempo -> Ingresar en la url el valor 
	http://tempo:3200, se usa el nombre del contenedor que almacena a tempo y su puerto ->
	save and test.
